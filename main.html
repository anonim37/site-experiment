Build and Test Conflicts in The Wild

Abstract:

Context: Collaborative software development allows developers to simultaneously contribute to the same project performing different activities. Although this might increase development productivity, it also brings conflicts among developers contributions. Conflicts can arise indifferent moments. For instance, during merge scenarios when different contributions are integrated (merge conflicts), as also after the integration, when the attempt to build the integration result fails due to compilation problems or failed tests (build and test conflicts, respectively). Goal: To understand how different contributions from merge scenarios influence on build and test conflicts occurrence, in this study we investigate the frequency, causes and adopted resolution patterns for these conflicts. Methodology: We perform an empirical study evaluating merge scenarios from Java projects using Travis CI for continuous integration and Maven as build manager. To identify conflicts, we access information from git repositories as also their associated build process. Filters were applied to establish merge commits that present broken build status, while their parent commits have superior ones (for example, a merge commit is not compilable while its parents are successful built). Besides parsing build logs for identifying the causes behind the broken builds, we also parse the source code to stablish the interferences between contributions. Different from previous studies, we evaluate the occurrence of conflicts from merge scenarios with merge conflicts leading us to classify our results based on contributor and integrator changes. Results: We identify most build and test conflicts happen due to contributor changes (especially in clean merge scenarios, without merge conflicts). In the other hand, integrator changes performed after contributions integration are also responsible for conflicts. Build conflicts are not restricted to compilation problems but also to static analysis (for example, verification of style conventions performed after the compilation process). The most recurrent causes concentrate on static semantic problems, especially Unavailable Symbol (the attempt to use an unavailable element in the project). In the same way, test conflicts involve not only failed tests but also failed analysis performed after tests execution (unachieved code coverage). In general, the applied resolution patterns adopt the conflicts to the new project state instead of returning to an old one. Conclusion: Our findings reinforce the evidence of build and test conflicts and also present information about the occurrence of conflicts considering scenarios from merge conflicts. This perspective leads us to evaluate how the way merge conflict fixes influence on build and test conflicts. For build conflicts, we define a catalog of causes that can be applied to assistive tools on software development aiming to avoid or treat such problems. Replication Study

During this work, we design a study involving three main steps as illustrated bellow. The first step is responsible for getting all information required for the study, which includes mining repositories (source code and builds information). The second step performs many verifications aiming to filter our data of false positive conflicts leaving only true ones. Finally, the last step identifies the causes of build and test conflicts and the resolution patterns adopted (identifying for each conflict the closest commit responsible for the fix).
